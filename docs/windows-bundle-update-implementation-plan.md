# Windows Bundle Update Implementation Plan

## 목적

Windows 트레이 앱의 번들 업데이트를 서비스 제어와 분리하고, macOS에서 이미 확정한 번들 업데이트 근간과 같은 논리로 정렬합니다. 명시적인 `앱 업데이트` 버튼을 통해서만 안전하게 앱 실행 파일 교체가 일어나도록 정리합니다.

Windows는 macOS와 달리 `.app` 번들이 아니라 self-contained single-file `.exe`를 게시하므로, 문서에서는 사용자 용어 일관성을 위해 `번들 업데이트`라고 부르되 실제 교체 대상은 실행 파일입니다.

이 문서의 핵심은 `앱 번들 교체만 책임지고 서비스 런타임 근간은 건드리지 않는다` 입니다.

## 공통 근간 로직

Windows도 아래 의미는 macOS와 동일해야 합니다.

- `서비스 시작`
  - 앱 제외 서비스 전체 새로 기동
  - 필요 시 `codex-adapter` 최신화
  - 서비스 재사용 없음
- `서비스 정지`
  - 앱 제외 OctOP 관련 프로세스 전부 종료
- `앱 업데이트`
  - 앱 번들 교체 전용
  - 서비스 런타임 업데이트와 분리
- `종료`
  - 서비스 정지 후 앱 종료

즉 Windows 번들 업데이트도 아래 불변 조건을 지켜야 합니다.

1. 서비스 런타임 원자적 업데이트 근간을 건드리지 않습니다.
2. 로그인 저장소 정책을 바꾸지 않습니다.
3. 서비스 시작/정지/종료 의미를 바꾸지 않습니다.
4. 서비스 런타임 포인터(`runtime-current`, `runtime-previous`)를 직접 수정하지 않습니다.
5. `stdio://` 정책을 바꾸지 않습니다.

## 저장소 경계

Windows도 저장소 경계는 macOS와 같은 논리를 유지해야 합니다.

- 로그인 데이터
  - 전역 `CODEX_HOME`
  - 기본 경로 `%USERPROFILE%\\.codex`
- 로그인 외 데이터
  - `%LOCALAPPDATA%\\OctOP`
  - 설정, 상태, 쓰레드/프로젝트/진단, 런타임, 캐시

금지:

- 로그인 데이터를 로컬 `codex-home`으로 복사하지 않음
- 전역 인증 파일을 번들 업데이트 백업/복구 대상으로 포함하지 않음
- 인증 데이터와 상태/쓰레드 데이터를 같은 저장소로 합치지 않음

## 앱/런타임 표기 규칙

Windows도 사용자에게 보이는 개념은 macOS와 맞춰야 합니다.

- 앱 버전
  - 현재 실행 중인 앱 번들 버전
- 런타임 ID
  - 활성 런타임의 `build-info.json.sourceRevision` 앞 12자리
- 런타임 업데이트 표시
  - 업데이트 없음: `런타임 ID <current12>`
  - 업데이트 있음: `런타임 ID <current12> · 업데이트 <remote12>`
- 앱 업데이트 버튼
  - 새 앱 번들이 있을 때만 노출
  - 일반 메뉴보다 강조

## 번들 업데이트의 역할

번들 업데이트가 해야 하는 일:

- 새 앱 번들 존재 여부 확인
- `앱 업데이트` 버튼 노출
- 업데이트 직전 서비스와 앱 프로세스 종료
- 기존 실행 파일 백업
- 새 실행 파일 설치
- 실패 시 기존 실행 파일 복원
- 성공 후 임시 산출물 정리

번들 업데이트가 하면 안 되는 일:

- `codex-adapter` 최신화 책임 가져가기
- 서비스 런타임 릴리즈 포인터 정책 바꾸기
- 서비스 시작/정지/종료 의미 바꾸기
- 로그인 저장소 위치 바꾸기
- 전역 인증 파일 복사/이동/삭제하기
- 서비스 런타임 원자적 업데이트 흐름을 우회하거나 대체하기
- 정상 경로에서 앱 로컬 데이터를 이동/복원하기

- 자동 업데이트가 `재시작` 경로와 섞여 있습니다.
- 현재 업데이트는 `.exe` 교체 중심이고, 성공/실패 판정과 정리 타이밍이 약합니다.
- 앱 로컬 데이터 취급 원칙이 문서/코드에서 충분히 고정돼 있지 않습니다.
- 인증 저장소 정책이 README와 코드 사이에서 일치하지 않습니다.
- 서비스 런타임 근간과 번들 업데이트의 경계가 문서/코드에서 충분히 고정되어 있지 않습니다.

## 목표 상태

### 메뉴/상태

- 최신 번들이 없으면 `앱 업데이트` 버튼을 숨깁니다.
- 최신 번들이 있으면 `앱 업데이트 <tag>` 버튼을 표시합니다.
- 버튼은 일반 메뉴보다 눈에 띄게 표시합니다.
- 버튼의 존재 여부는 서비스 런타임 업데이트 표시와 별개입니다.

### 업데이트 실행

1. 사용자가 `앱 업데이트`를 누릅니다.
2. 앱은 현재 정의된 `서비스 정지`를 수행합니다.
3. 외부 업데이트 프로세스가 앱 종료를 기다립니다.
4. 외부 스크립트가 서비스 관련 프로세스와 포트 해제를 재검증합니다.
5. 현재 실행 파일을 `.previous-update` 로 백업합니다.
6. 새 실행 파일을 설치 위치에 배치합니다.
7. 새 앱을 실행합니다.
8. 새 앱 launch 확인 신호가 생길 때까지 대기합니다.

### 실패 복구

- 설치 실패 시:
  - 기존 실행 파일 즉시 복원
  - 이전 앱 자동 실행
- 전역 인증 저장소는 건드리지 않습니다.
- 정상 경로에서는 앱 로컬 데이터를 복원하지 않습니다.

### 성공 후 정리

- 새 앱이 정상적으로 기동하면:
  - `.previous-update`
  - 백업 메타 폴더
  - 임시 다운로드 폴더
  를 정리합니다.

## 번들 업데이트가 지켜야 할 불변 조건

1. 서비스 런타임 릴리즈 구조 유지
   - `runtime-releases`
   - `runtime-current.txt`
   - `runtime-previous.txt`
2. 런타임 업데이트 감지 정책 유지
   - `codex-adapter` 내용 기준 감지
   - 런타임 ID/업데이트 표시 정책 유지
3. 로그인 저장소 정책 유지
   - 전역 `CODEX_HOME` 직접 참조
4. 서비스 시작/정지/종료 의미 유지
5. `stdio://` 세션 정책 유지
   - 서비스 판정에는 제외
   - 종료/업데이트 전 정리 대상에는 포함

## 데이터 취급 원칙

정상 Windows 번들 업데이트에서는 아래 원칙을 따릅니다.

- 앱 로컬 데이터는 `%LOCALAPPDATA%\\OctOP` 아래에 그대로 둡니다.
- 새 실행 파일 설치를 위해 데이터를 다른 위치로 옮길 필요가 없습니다.
- 정상 성공 경로에서는 데이터 복원 단계가 없습니다.
- 데이터 복구는 예외 상황에서만 별도 재해복구 절차로 수행합니다.

정상 업데이트 경로에서 건드리지 않아야 할 데이터:

- `config.json`
- `bridge-id.txt`
- `pending-login.json`
- `pending-service-start`
- `state`
- `runtime-current.txt`
- `runtime-previous.txt`
- `runtime-releases`
- `%USERPROFILE%\\.codex`
- 외부 `CODEX_HOME`

## 백업 구조

예시:

```text
%LOCALAPPDATA%\OctOP.update-backup\
  exe\
    OctOP.WindowsAgentMenu.exe.previous-update
  meta\
    target-version.txt
    started-at.txt
    status.json
    apply-update.log
```

정상 경로에서는 앱 로컬 데이터 전체 백업을 강제하지 않습니다.

## 업데이트 스크립트 요구사항

외부 PowerShell 스크립트는 앱 종료 후에도 계속 실행돼야 합니다.

필수 단계:

1. 기존 앱 PID 종료 대기
2. 서비스 관련 프로세스 잔존 여부 확인
3. 서비스 포트 해제 여부 확인
4. 기존 백업 폴더 정리 또는 충돌 방지
5. 현재 실행 파일 백업
6. 새 실행 파일 설치
7. 실패 시 실행 파일 롤백
8. 성공 시 새 앱 실행
9. 새 앱 launch 확인 신호 대기
10. 확인 후 임시 산출물 정리

중요:

- 이 스크립트는 서비스 런타임 포인터를 직접 수정하지 않습니다.
- 새 앱이 실행된 후 스스로 기존 서비스 시작/정지/런타임 갱신 로직을 수행하게 둡니다.

## 포트/프로세스 확인 원칙

업데이트 직전 정리 대상:

- `run-local-agent.mjs`
- `run-bridge.mjs`
- `services\\codex-adapter\\src\\index.js`
- `codex app-server --listen ws://...`
- 필요 시 `codex app-server --listen stdio://`

포트 확인 대상:

- 브릿지 포트
- 서비스용 WS app-server 포트

원칙:

- 앱은 먼저 `서비스 정지`를 수행합니다.
- 외부 업데이트 스크립트는 그 결과를 재검증합니다.
- 프로세스/포트 종료 확인 전에는 실행 파일 교체를 시작하면 안 됩니다.
- 시작 시 광범위한 포트 강제 종료로 대체하면 안 됩니다.

## 실패 시 롤백 원칙

아래 중 하나라도 실패하면 롤백합니다.

- 새 실행 파일 설치 실패
- 새 앱 실행 실패
- 새 앱 첫 실행 확인 실패

롤백 절차:

1. 새로 설치한 실행 파일 제거
2. 백업 실행 파일 원위치 복원
3. 이전 앱 실행
4. 롤백 실패 시 사용자에게 명확한 진단 로그 남김

## 성공 판정 원칙

다음이 만족되면 성공으로 봅니다.

- 새 실행 파일이 설치 위치에서 실행됨
- 새 앱이 시작됨
- 기존 로컬 데이터가 손대지 않은 상태로 유지됨
- 복구 대상 백업 폴더가 더 이상 필요하지 않음

## 구현 전제조건

아래는 구현 항목이 아니라 항상 유지돼야 하는 전제조건입니다.

1. 서비스 런타임 원자적 업데이트 근간을 건드리지 않는다.
2. 로그인 데이터는 전역, 그 외 데이터는 로컬이라는 저장소 경계를 바꾸지 않는다.
3. `서비스 시작`, `서비스 정지`, `앱 업데이트`, `종료` 의미를 바꾸지 않는다.
4. 번들 업데이트는 실행 파일 교체만 담당하고, 런타임 업데이트 책임을 가져가지 않는다.
5. `stdio://` 세션 정책을 바꾸지 않는다.
