# Bundle Update Implementation Plan

## 목적

이 문서는 macOS `앱 번들 업데이트`의 구현 계획서입니다.

가장 중요한 전제는 다음입니다.

- 서비스 런타임 원자적 업데이트의 근간은 [service-runtime-atomic-update-plan.md](/Users/jazzlife/Documents/Workspaces/Products/OctOP/docs/service-runtime-atomic-update-plan.md) 가 담당합니다.
- 번들 업데이트는 그 근간을 건드리지 않고, `앱 번들 교체` 역할만 충실히 수행해야 합니다.

즉 이 문서의 목적은:

- 앱 번들 교체를 명시적인 `앱 업데이트` 버튼으로만 수행하고
- 서비스 런타임, 인증 저장소, 서비스 시작/정지 의미를 흔들지 않으며
- 실패 시 이전 앱 번들로 안전하게 되돌아가게 만드는 것입니다.

## 번들 업데이트의 역할

번들 업데이트가 해야 하는 일:

- 새 앱 번들 존재 여부 확인
- `앱 업데이트` 버튼 노출
- 업데이트 직전 서비스와 앱 프로세스 종료
- 기존 앱 번들 백업
- 앱 로컬 데이터 백업
- 새 앱 번들 설치
- 필요 시 앱 로컬 데이터 복구
- 실패 시 기존 앱 번들 복원
- 성공 후 임시 산출물 정리

번들 업데이트가 하면 안 되는 일:

- `codex-adapter` 최신화 책임 가져가기
- 서비스 런타임 릴리즈 포인터(`current-release.txt`, `previous-release.txt`) 정책 바꾸기
- 서비스 시작/정지/종료 의미 바꾸기
- 로그인 저장소 위치 바꾸기
- 전역 인증 파일 복사/이동/삭제하기
- 서비스 런타임 원자적 업데이트 흐름을 우회하거나 대체하기

## 서비스 런타임과의 경계

이 문서는 아래 사항을 전제로 합니다.

- `서비스 시작`
  - 앱 제외 서비스 전체 새로 기동
  - 필요 시 `codex-adapter` 최신화
  - 서비스 재사용 없음
- `서비스 정지`
  - 앱 제외 OctOP 관련 프로세스 전부 종료
- `종료`
  - 서비스 정지 후 앱 종료
- `앱 업데이트`
  - 앱 번들 교체 전용

즉 번들 업데이트는 `서비스 시작` 로직을 다시 설계하지 않습니다.
업데이트 직전에는 기존 의미의 `서비스 정지`를 호출해 서비스만 정리하고, 그 다음 앱 번들을 교체합니다.

## 저장소 경계

번들 업데이트는 저장소 경계를 절대 흐리면 안 됩니다.

### 전역 저장소

- 로그인 데이터:
  - `CODEX_HOME`
  - 기본 경로 `~/.codex`
- 현재 원칙:
  - 인증은 전역 저장소를 직접 참조
  - 앱 로컬 저장소로 복사 금지

### 로컬 저장소

- 앱 로컬 루트:
  - `~/Library/Application Support/OctOPAgentMenu`
- 포함 데이터:
  - 설정
  - 상태
  - 쓰레드/프로젝트/진단
  - 런타임 캐시 및 릴리즈 디렉터리

### 번들 업데이트 원칙

- 백업/복원 대상:
  - 앱 번들
  - 앱 로컬 저장소
- 명시적 제외 대상:
  - `~/.codex`
  - `~/.codex/auth.json`
  - 전역 인증 관련 파일

## 현재 구현 기준 사실

현재 코드에서 이미 구현된 부분:

- `앱 업데이트` 버튼이 별도 메뉴로 존재함
- 버튼은 새 버전이 있을 때만 노출됨
- 최신 버전 확인은 GitHub `tags` API + 실제 zip 자산 존재 여부 확인으로 수행함
- 업데이트 전 서비스 정지 후 앱 종료 대기, 런타임 프로세스 종료, 브릿지/WS 포트 해제 확인을 수행함
- 앱 로컬 데이터는 `~/Library/Application Support/OctOPAgentMenu.update-backup` 으로 전체 백업함
- 기존 앱 번들은 `<CurrentApp>.previous-update` 로 백업함
- 앱 데이터 백업 루트에 `status.json` 을 기록해 대상 태그, 현재 앱 경로, launch 확인 시각을 남김
- 앱 데이터 백업 루트에 `apply-update.log` 를 기록해 외부 교체 스크립트 단계별 결과를 남김
- 새 앱은 시작 직후 launch 확인 마커를 남기고, 이 마커가 생기기 전에는 백업 정리를 보류함
- 교체 스크립트는 새 앱 실행 후 launch 확인 마커가 생길 때까지 대기하고, 실패 시 이전 앱 번들을 즉시 복원함
- 테스트에서는 `OCTOP_AGENT_MENU_APP_UPDATE_TAGS_URL`, `OCTOP_AGENT_MENU_APP_UPDATE_ASSET_BASE_URL`, `OCTOP_AGENT_MENU_APP_UPDATE_FORCE_ENABLED` override 로
  태그/자산 존재 상황을 로컬 fixture 로 재현할 수 있음
- 번들 업데이트 준비 단계는 `prepareAvailableAppUpdate(...)` 로 분리돼, 실제 적용 직전까지의 경로를 테스트 코드가 그대로 검증할 수 있음

현재 코드에서 아직 약한 부분:

- 데이터 롤백은 앱 번들 롤백만큼 강하게 구현돼 있지 않음
- 데이터 복구는 다음 실행 시 `appSupportURL`이 없을 때만 복구하는 보수적 경로임
- 실제 번들 교체를 끝까지 수행하는 실환경 E2E 검증은 아직 남아 있음

즉 이 문서는 현재 구조를 기준으로 다음 구현 방향을 정리해야 합니다.

## 핵심 요구사항

1. 번들 업데이트는 서비스 런타임 원자적 업데이트와 역할이 겹치면 안 됩니다.
2. `앱 업데이트` 버튼은 새 앱 번들이 있을 때만 보여야 합니다.
3. 새 앱 번들 존재 판정은:
   - 원격 태그 존재
   - 해당 태그의 macOS zip 자산 존재
   를 둘 다 만족해야 합니다.
4. 업데이트 시작 전 서비스와 앱 프로세스, 서비스 포트는 완전히 해제돼야 합니다.
5. 전역 인증 저장소는 번들 업데이트 과정에서 건드리면 안 됩니다.
6. 실패 시 최소한 기존 앱 번들은 즉시 복원돼야 합니다.
7. 성공 후에는 불필요한 백업과 임시 산출물을 정리해야 합니다.

## 목표 상태

### 메뉴/상태

- 최신 앱 번들이 없으면 `앱 업데이트` 버튼을 숨깁니다.
- 최신 앱 번들이 있으면 `앱 업데이트 <tag>` 버튼을 표시합니다.
- 버튼은 일반 메뉴보다 눈에 띄게 표시합니다.
- 버튼의 존재 여부는 서비스 런타임 업데이트 표시와 별개입니다.

### 업데이트 실행

1. 사용자가 `앱 업데이트`를 누릅니다.
2. 앱은 현재 정의된 `서비스 정지`를 수행합니다.
3. 앱은 서비스가 실제로 내려갔는지 기다립니다.
4. 외부 업데이트 스크립트가 앱 종료를 기다립니다.
5. 외부 스크립트가 런타임 관련 프로세스와 서비스 포트 해제를 재검증합니다.
6. 현재 앱 번들을 `.previous-update` 로 이동합니다.
7. 앱 로컬 데이터를 `.update-backup` 으로 백업합니다.
8. 새 앱 번들을 설치 위치에 배치합니다.
9. 새 앱을 실행합니다.
10. 새 앱이 launch 확인 마커를 남길 때까지 대기합니다.

### 실패 복구

- 앱 번들 설치 실패 시:
  - 기존 앱 번들을 즉시 복원
  - 이전 앱 자동 실행
- 앱 데이터 복구가 필요한 경우:
  - 앱 로컬 저장소 백업에서 복원
  - 단, 전역 인증 저장소는 건드리지 않음

### 성공 후 정리

- 새 앱이 정상적으로 기동하면:
  - `.previous-update`
  - `.update-backup`
  - `.update-backup.staging`
  - 임시 다운로드/압축 해제 폴더
  를 정리합니다.

현재 구현에서는 새 앱이 `status.json` 의 `currentAppPath` 와 일치하는 번들 경로로 뜬 뒤
`launchConfirmedAt` 과 `launch-confirmed` 마커를 기록해야만 정리를 진행합니다.

## 번들 업데이트가 지켜야 할 불변 조건

아래 조건은 번들 업데이트 구현이 바뀌어도 유지돼야 합니다.

1. 서비스 런타임 릴리즈 구조 유지
   - `runtime/releases/<id>`
   - `runtime/current-release.txt`
   - `runtime/previous-release.txt`
2. 런타임 업데이트 감지 정책 유지
   - 원격 `codex-adapter` 리비전 비교
   - 메뉴의 런타임 ID/업데이트 표시 정책 유지
3. 로그인 저장소 정책 유지
   - 전역 `~/.codex` 직접 참조
4. 서비스 시작/정지/종료 의미 유지
5. `stdio://` 세션 정책 유지
   - 서비스 판정에는 제외
   - 종료/업데이트 전 정리 대상에는 포함

## 제안 구현 구조

### 1. 업데이트 상태 조회 계층

`AgentBootstrapStore`에서 유지할 책임:

- 현재 앱 버전
- 최신 앱 버전 태그
- 업데이트 가능 여부
- 업데이트 확인 진행 상태
- 마지막 확인 오류

현재 구현 기반 함수:

- `refreshAvailableAppUpdate(...)`
- `availableAppUpdate`
- `hasAvailableAppUpdate`

이 계층은 서비스 런타임 업데이트 상태와 독립적으로 유지합니다.

### 2. 메뉴 계층

`OctOPAgentMenuApp.swift`에서 유지할 책임:

- `앱 업데이트` 버튼 노출/숨김
- 버튼 라벨에 새 버전 표시
- 업데이트 실행 중 비활성화
- 서비스 버튼과 역할 분리

금지:

- `재시작`이나 `서비스 시작`에 번들 교체 책임을 다시 넣지 않음

### 3. 업데이트 실행 계층

`AgentUpdateSupport.swift`에서 유지할 책임:

- 태그 조회
- 자산 존재 확인
- 다운로드
- 압축 해제
- 교체 스크립트 생성
- 교체 스크립트 실행

이 계층이 추가로 보장해야 하는 것:

- 앱 종료 대기
- 런타임 프로세스 종료 재검증
- 브릿지/WS 포트 해제 재검증
- 앱 번들 롤백

### 4. 데이터 백업/복원 계층

`AgentBootstrapSupport.swift`에서 유지할 책임:

- 앱 로컬 데이터 전체 백업
- 필요 시 앱 로컬 데이터 복구
- 성공 후 백업 정리

중요:

- 이 계층은 서비스 런타임 포인터 구조를 재설계하지 않습니다.
- `runtime` 디렉터리도 앱 로컬 데이터의 일부로 다루되, 그 내부 구조 정책은 원자적 업데이트 문서를 따릅니다.

## 백업 대상

최소 백업 대상:

- 현재 앱 번들
- `~/Library/Application Support/OctOPAgentMenu` 전체

명시적 제외:

- `~/.codex`
- `~/.codex/auth.json`
- 전역 인증 관련 파일

## 실제 백업 경로

현재 구현 기준:

- 앱 번들 백업:
  - `<CurrentApp>.previous-update`
- 앱 데이터 백업:
  - `~/Library/Application Support/OctOPAgentMenu.update-backup`
- staging 백업:
  - `~/Library/Application Support/OctOPAgentMenu.update-backup.staging`
- 임시 다운로드 루트:
  - `NSTemporaryDirectory()/OctOPAgentMenu/updates/<tag>`

## 업데이트 스크립트 요구사항

외부 스크립트는 앱 종료 후에도 계속 실행돼야 합니다.

필수 단계:

1. 기존 앱 PID 종료 대기
2. 런타임 경로 기반 프로세스 잔존 여부 확인
3. 브릿지 포트 해제 확인
4. WS app-server 포트 해제 확인
5. 기존 `.previous-update` 정리
6. 현재 앱 번들을 `.previous-update` 로 이동
7. 새 앱 번들을 설치 위치에 복사
8. 실패 시 기존 앱 번들 복원
9. 성공 시 새 앱 실행
10. 임시 업데이트 폴더 정리

중요:

- 이 스크립트는 서비스 런타임 릴리즈 포인터를 직접 수정하지 않습니다.
- 새 앱이 이후 스스로 서비스 시작/정지/런타임 갱신 로직을 수행하게 둡니다.

## 포트/프로세스 확인 원칙

업데이트 직전 정리 대상:

- `run-local-agent.mjs`
- `run-bridge.mjs`
- `services/codex-adapter/src/index.js`
- `codex app-server --listen ws://...`
- 필요 시 `codex app-server --listen stdio://`

포트 확인 대상:

- 브릿지 포트
- 서비스용 WS app-server 포트

원칙:

- 서비스 정지는 앱 쪽에서 먼저 수행
- 외부 업데이트 스크립트는 종료 결과만 재검증
- 번들 업데이트가 서비스 종료 정책 자체를 재정의하지 않음

## 실패 시 롤백 원칙

필수 롤백:

- 새 앱 번들 설치 실패
- 앱 번들 배치 후 실행 전 오류

권장 롤백:

- 새 앱 첫 실행 직후 치명적 실패가 감지되는 경우

현재 문서 기준으로 우선 보장해야 하는 최소 롤백:

1. 새 앱 번들 제거
2. `.previous-update` 앱 번들 원위치 복원
3. 이전 앱 실행

앱 데이터 롤백은 앱 번들 롤백과 별도로 다루며, 전역 인증 저장소는 제외합니다.

## 성공 판정 원칙

번들 업데이트의 성공 판정은 번들 관점으로만 정의합니다.

필수 성공 조건:

- 새 앱 번들이 설치 위치에 존재함
- 새 앱이 실행됨

추가 확인 조건:

- 앱 로컬 데이터 루트가 유지됨
- 다음 실행에서 백업 정리 루틴이 정상 동작함

주의:

- 서비스 런타임이 최신 `codex-adapter`로 올라가는지는 이 문서의 직접 성공 조건이 아닙니다.
- 그 책임은 서비스 런타임 원자적 업데이트 문서가 담당합니다.

## 구현 전제조건
